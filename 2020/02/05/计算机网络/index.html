计算机网络

  • 网络进程标识 -> IP + port

  • 应用层协议大致分为下面几个方面。

  1. 消息的类型(type)
  2. 消息的语法(syntax)/格式
  3. 字段的语义(semantics)
  4. 规则(rules)

网络应用对传输服务的需求

  1. 数据丢失(data loss) / 可靠性（reliability)

  2. 时间(timing) / 延迟 (delay)

  3. 带款(bandwidth)

  • 不局限这三种需求，但是这三种是最主要的。

  • URL(Uniform Resoure Locator) : 统一资源定位器

  • Scheme://host:port/path -> 标准格式

响应时间分析与建模

  • RTT(Round Trip Time)
  • 从客户端发送一个很小的数据包到服务器并返回所经历的时间。
  1. 非持久性连接。

  • 发起，建立TCP连接：1个RTT。

  • 发送HTTP请求到HTTP响应消息前几个字节到达: 1个RTT。

  • 响应消息中文件/对象传输时间。

  • 所以，total: 2RTT + 文件发送时间。

  • 存在问题: 资源占用多(每个TCP连接操作系统都会为其分配资源),时间(浏览器因为会优化所以会并发多个TCP连接，给服务器带来很大的影响)

  1. 持久性连接
  • 服务器发送TCP响应后，一直保持连接，直到所有HTTP消息传输完。
  • 其中又可以分为无流水的持久性连接: 客户端只有收到前一个响应后才发送新的请求。和带有流水机制的持久性连接: 客户端只要遇到一个引用对象就尽快发出请求，像生产工厂的流水线一样。

拓展: 客户端向服务端传输数据的方法

  1. POST方法: 提交表格 , 在请求消息的消息体(Entity body)中上传客户端的输入。
  2. URL方法: 即GET方法 , 输入的信息通过request行的URL字段上传。例如下:
  • http://www.somesite.com/animalsearch?**monkeys&banana**(这个是随即乱编的url)




  • http/1.0 -> GET / POST / HEAD: 请Server不要将所请求的对象放入响应消息中。(一般做测试用)

  • http/1.1 -> GET / POST / HEAD / PUT : 消息体文件上传到URL字段指定的路径中. / DELETE : 删除URL字段所指定的文件
