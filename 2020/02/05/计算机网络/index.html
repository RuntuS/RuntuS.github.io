计算机网络

  • 网络进程标识 -> IP + port

  • 应用层协议大致分为下面几个方面。

  1. 消息的类型(type)
  2. 消息的语法(syntax)/格式
  3. 字段的语义(semantics)
  4. 规则(rules)

网络应用对传输服务的需求

  1. 数据丢失(data loss) / 可靠性（reliability)

  2. 时间(timing) / 延迟 (delay)

  3. 带款(bandwidth)

  • 不局限这三种需求，但是这三种是最主要的。

  • URL(Uniform Resoure Locator) : 统一资源定位器

  • Scheme://host:port/path -> 标准格式

响应时间分析与建模

  • RTT(Round Trip Time)
  • 从客户端发送一个很小的数据包到服务器并返回所经历的时间。
  1. 非持久性连接。

  • 发起，建立TCP连接：1个RTT。

  • 发送HTTP请求到HTTP响应消息前几个字节到达: 1个RTT。

  • 响应消息中文件/对象传输时间。

  • 所以，total: 2RTT + 文件发送时间。

  • 存在问题: 资源占用多(每个TCP连接操作系统都会为其分配资源),时间(浏览器因为会优化所以会并发多个TCP连接，给服务器带来很大的影响)

  1. 持久性连接
  • 服务器发送TCP响应后，一直保持连接，直到所有HTTP消息传输完。
  • 其中又可以分为无流水的持久性连接: 客户端只有收到前一个响应后才发送新的请求。和带有流水机制的持久性连接: 客户端只要遇到一个引用对象就尽快发出请求，像生产工厂的流水线一样。

拓展: 客户端向服务端传输数据的方法

  1. POST方法: 提交表格 , 在请求消息的消息体(Entity body)中上传客户端的输入。
  2. URL方法: 即GET方法 , 输入的信息通过request行的URL字段上传。例如下:
  • http://www.somesite.com/animalsearch?**monkeys&banana**(这个是随即乱编的url)




  • http/1.0 -> GET / POST / HEAD: 请Server不要将所请求的对象放入响应消息中。(一般做测试用)

  • http/1.1 -> GET / POST / HEAD / PUT : 消息体文件上传到URL字段指定的路径中. / DELETE : 删除URL字段所指定的文件

Web缓存/代理服务器技术

  • 功能：在不妨问服务器的前提下满足客户端的http请求。
  • 这个技术主要是用于性能优化的。
  • 设置一个缓存服务器，客户端的请求发送至该缓存服务器，如果服务器有对应的响应报文，则直接发回给客户端；如果没有该报文，则直接去真正服务器的地方取对应响应消息。从而提高了效率，降低了带宽压力。

    代理服务器怎么直到自己缓存的响应消息是最新的？ -> 条件GET方法

  • proxy通过if-modefied-since头部行去和真正server端的对应消息的对应字段进行比对，如果是最新的，则server端的响应消息不包含对象（304 Not Modified）。如果不是最新的，则会把最新的内容通过响应体发送回来。

DNS(Domain Name System)应用

  • 解决主机/路由器的识别问题。(IP地址/域名)
  • DNS是一个分布式的分层的数据库。

  • IP地址和域名之间映射问题
  • 需要使用域名解析系统(DNS)来解决映射问题(把域名解析为IP地址)
  • DNS是Internet的核心功能，用应用层协议实现。

DNS服务

  • 域名 -> IP地址 // 主机别名 // 邮件服务器别名 // 负载均衡: Web服务器

域名服务器的大概层次(因为是分层的)

  • 根域名服务器(全球13个) > 顶级服务器(TLD,top-level domain): 负责com,org,net,edu等顶级域名和国家域名。 > 权威域名服务器: 组织的域名解析服务器，提供组织内部服务器的解析服务

  • 注意本地域名解析服务器不严格属于层级体系

  • 每个ISP有一个本地域名服务器(默认域名解析服务器) ， 当主机进行DNS查询时，查询被发送到本地域名服务器(在电脑上可以自己指定本地域名服务器): 即本地域名服务器可以做一个代理服务器(proxy)。
